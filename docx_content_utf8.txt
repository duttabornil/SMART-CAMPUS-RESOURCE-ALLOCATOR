ABSTRACT
The Smart Campus Resource Allocation System is a database-oriented application designed to efficiently manage and allocate campus resources such as classrooms, laboratories, seminar halls, and projectors. In many colleges, improper scheduling and lack of a centralized systemlead to resource conflicts, double bookings, and inefficient utilization of available facilities. This project aims to solve these problems by applying fundamental and advanced Database Management System (DBMS) concepts studied in the fourth semester. The system uses a relational database model to store and manage data related to resources, departments, users, bookings, and usage logs. Core DBMS concepts such as entity-relationship modelling, normalization, primary keys, and foreign keys are implemented to ensure data integrity and eliminate redundancy. Conflict-free scheduling is achieved using SQL queries and database triggers that prevent overlapping bookings of the same resource. An administrative approval mechanism is included to demonstrate role-based access control, where authorized users can approve or reject booking requests. The system also incorporates database views and indexing techniques to generate usage reports and improve query performance. These reports help in analysing resource usage patterns and understanding how data can be used for decision-making. Through this project, students gain practical experience in database design, SQL (DDL and DML) commands, constraints, triggers, joins, and transactions. The Smart Campus Resource Allocation System not only improves campus resource management but also serves as an effective learning platform for understanding real-world applications of DBMS concepts.

 Introduction
Educational institutions rely heavily on shared resources such as classrooms, laboratories, seminar halls, and projectors for daily academic activities. Efficient allocation of these resources is essential to ensure smooth functioning of lectures, events, and practical sessions. However, traditional booking methods such as manual registers, spreadsheets, or informal communication often led to errors and conflicts.
The Smart Campus Resource Allocation System is a database-oriented application designed to automate and streamline resource scheduling. It provides a centralized platform where users can check availability, request bookings, and track usage while administrators oversee approvals and management.
 Motivation
The motivation for this project arises from:
Frequent double bookings of classrooms and labs
Manual and time-consuming allocation processes
Lack of transparency in resource availability
Inefficient utilization of campus facilities
Difficulty in tracking usage statistics
This project aims to create a structured and automated system that improves accuracy, efficiency, and accountability in campus resource management while implementing real-world DBMS concepts.
 Scope
The system supports:
User registration and authentication
Resource listing and categorization
Online booking requests
Conflict-free scheduling
Administrative approval mechanism
Usage reporting and analytics
Secure database management
The system is designed for students, faculty members, and administrators within an institution.
 Problem Statement
Most educational institutions lack a centralized system to manage shared resources, resulting in scheduling conflicts, inefficient usage, and manual errors. The objective of this project is to design and implement a database-driven system that manages resources, automates booking processes, and ensures reliable, conflict-free allocation using structured DBMS techniques.

3.1 Adding Constraints and Queries Based on Constraints
Constraints are used in database systems to maintain data integrity and ensure that invalid or inconsistent data cannot be inserted into the tables. In the Smart Campus Resource Allocation System, constraints are applied to ensure unique records, maintain relationships between tables, and enforce valid data ranges.

3.2 Queries Based on Aggregate Functions
Aggregate functions are used to perform calculations on multiple records and return a single summarized value.
Question 1
Find the total number of bookings for each resource.
SQL Statement

OutputDisplays the number of bookings for each resource.
Question 2
Find the average capacity of resources in each department.
SQL Statement

OutputDisplays the average resource capacity per department.
Question 3
Find the total number of users in each department.
SQL Statement

OutputDisplays the total number of users belonging to each department.

Set operations are used to combine results from multiple queries.

4.4 Third Normal Form (3NF)
4.4.1: Identify Dependency
USERS
Direct Dependency:User_ID → Name, Email, Department_ID
Since User_ID uniquely identifies each user, all user-related attributes depend directly on it. There is no dependency of non-key attributes on other non-key attributes within this table after separation.
DEPARTMENT
Direct Dependency:Department_ID → Department_Name
Each department is uniquely identified by Department_ID, and the department name depends directly on it. No transitive dependency exists within this table.
RESOURCE
Direct Dependency:Resource_ID → Resource_Name, Resource_Type, Location, Capacity, Department_ID
All attributes of the resource depend directly on the primary key Resource_ID. There are no non-key attributes determining other attributes.
BOOKING
Direct Dependency:Booking_ID → Booking_Date, Start_Time, End_Time, Status, User_ID, Resource_ID
Each booking is uniquely identified by Booking_ID, and all related attributes depend directly on it. There is no transitive dependency present.
BOOKING_SLOT
Direct Dependency:(Booking_ID, Slot_No) → Slot_Start_Time, Slot_End_Time
The combination of Booking_ID and Slot_No uniquely identifies each slot. Slot timings depend directly on this composite key.
USAGE_LOG
Direct Dependency:Log_ID → Booking_ID, Actual_Start_Time, Actual_End_Time, Usage_Duration
Each log entry is uniquely identified by Log_ID, and all attributes depend directly on it without any transitive dependency.
Transitive Dependency Identified
User_ID → Department_ID → Department_Name
Here, Department_Name depends on Department_ID, which in turn depends on User_ID. This creates a transitive dependency, which violates Third Normal Form (3NF).

CHAPTER 5
IMPLEMENTATION OF CONCURRENCY CONTROL AND RECOVERY MECHANISMS
 5.1 Introduction to Transactions
In a database system, multiple users may access and manipulate data simultaneously. In the Smart Campus Resource Allocation System, users such as students, faculty members, and administrators can perform operations like booking resources, updating schedules, and viewing availability at the same time. This concurrent access may lead to data inconsistency and conflicts if not properly managed.
5.1.1 Properties
1. Atomicity
Atomicity ensures that a transaction is treated as a single unit of work. Either all operations of a transaction are completed successfully, or none of them are applied.
2. Consistency
Consistency ensures that a transaction takes the database from one valid state to another valid state. All integrity constraints such as primary keys and foreign keys must be maintained
3. Isolation
Isolation ensures that multiple transactions executing at the same time do not interfere with each other. Each transaction is executed as if it is the only transaction in the system.
4. Durability
Durability ensures that once a transaction is committed, the changes made are permanent, even in case of system failure.
5.1.2 States
A transaction passes through different states during its execution in a database system.
Active
The transaction is currently executing and performing read/write operations.
Partially Committed
The transaction has completed all its operations but the changes are not yet permanently saved.
Committed
The transaction has been successfully completed and all changes are permanently stored in the database.
Failed
The transaction cannot proceed further due to an error or system failure.
Aborted
The transaction is rolled back, and all changes made by it are undone.
Terminated
The transaction has finished execution, either after commit or abort, and releases all resources.
5.2 Transaction Control Language (TCL)
 TCL commands are used to control transactions in the database.
5.2.1 Save point
A savepoint creates a temporary checkpoint inside the transaction. 
SAVEPOINT sp1; 
Example in project: Creates a restore point after booking insertion so that if slot insertion fails, the system can rollback only the slot entries without affecting the booking.
5.2.2Commit
COMMIT permanently saves all changes made during the transaction. 
COMMIT; 
5.2.3 Rollback 
ROLLBACK undoes all changes made during the current transaction. 
ROLLBACK;

This transaction demonstrates inserting a booking and slot details using a savepoint. If an error occurs, rollback is performed to the savepoint.

-- Rollback to savepoint (removes slot entries only)
ROLLBACK TO sp1;
-- Commit booking
COMMIT;

5.3 Concurrency control 
Concurrency control ensures that multiple users can access the database simultaneously without causing inconsistency or data conflicts. It is mainly used to prevent issues such as lost updates, dirty reads, and inconsistent data when more than one transaction tries to access the same data at the same time.
Concurrency control in this project ensures that multiple users can book resources simultaneously without causing conflicts or data inconsistency.
5.3.1 Concurrency control Algorithms 
5.3.1 Locking commands
a. Row-Level Locking – SELECT ... FOR UPDATE
Locks specific rows to prevent other transactions from modifying them
SELECT *FROM BookingWHERE Booking_ID = 301FOR UPDATE;
Locks only the selected booking record
 Other users cannot update/delete this booking until the transaction completes
 Used to prevent double booking of the same resource
b. Table-Level Locking – LOCK TABLE
Locks the entire table
LOCK TABLE Booking IN EXCLUSIVE MODE;
No other transaction can read or write to the Booking table
 Used during critical operations like bulk updates or maintenance
Lock Mode
Description
ROW SHARE
Allows concurrent access; prevents exclusive table locks
ROW EXCLUSIVE
Used for INSERT/UPDATE/DELETE; restricts other locks
SHARE
Allows only read operations; blocks updates
SHARE ROW EXCLUSIVE
Allows reads but restricts other locks
EXCLUSIVE
Prevents all other access (full table lock)
c. COMMIT – Release All Locks
After successful execution of the transaction, COMMIT is used to permanently save the changes and release all acquired locks.
COMMIT;
In our project, once a booking is successfully inserted or updated, COMMIT saves the changes and releases the locks on the Booking table.
d. ROLLBACK – Undo Changes & Release Locks
ROLLBACK is used when an error occurs during the transaction. It undoes all changes made during the current transaction and releases all locks.
ROLLBACK;
This ensures data consistency in case of transaction failure, such as an invalid booking or conflict in resource allocation.
